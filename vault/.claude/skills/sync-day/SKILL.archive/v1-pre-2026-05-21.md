# Skill: Sync Day (mid-day reconciliation)

Mid-day bidirectional sync — keeps Day Planner ↔ Kanban state continuous **without waiting for EOD `/daily-note --close`**. Captures ad-hoc items, surfaces Kanban tasks due today, updates daily note frontmatter live。

> **Why this exists**: `/daily-note --close` Phase 7 是 EOD-only sync。一天中临时 Day Planner 上插入的 ad-hoc tasks 完成后不入 Kanban → 多日 chain / 忘记 close 时 drift 累积。`/sync-day` 是 mid-day 入口。详 [[I260514-mid-day-sync-prevents-eod-reconciliation-drift]]。

## When to Use

- User says "/sync-day" / "sync day" / "今天同步一下" / "Day Planner 同步 Kanban" / "对一下今天进度"
- Mid-day check-in (建议 12:00 / 18:00 各 1 次)
- After a busy morning with many ad-hoc inserts
- 长 chain (出差日 / 忘记 close 多日) — sync 不堆积
- Auto-chain candidate: after `/inbox-process` or `/replan`

**Don't use for:**
- EOD 完整闭日 → `/daily-note --close` Phase 7
- 新建明天 daily → `/daily-note`
- 创建单个 task → `/task-capture`
- 重排 Day Planner 整体 → `/replan`

## How It Works

### Phase 1 — Resolve target + Parse today's Day Planner

```bash
today=$(date +%Y-%m-%d)
daily_note="04 Notes/daily notes/${today}.md"
```

如 daily note 不存在 → 停 → "今天 daily note 还没建，先 `/daily-note`。"

Read daily note `## Day Planner` section. Parse each line matching `^- \[(.)\] (\d{2}:\d{2}) (.+?)(?:\s*\{\{datetimeStart::.*\}\})?$`:
- Checkbox state (`[ ]` / `[x]` / `[/]` / `[-]`)
- Time `HH:mm`
- Task body (strip trailing `{{datetimeStart:: ...}}` if present — v3 hybrid format since 2026-05-21)
- Skip `BREAK` and `END` lines

Capture into 3 buckets:
- **bucket-A**: `[x]` items (completed, candidates for Kanban capture)
- **bucket-B**: `[ ]` items (still pending — may need rescheduling at EOD or push to tomorrow carry-forward)
- **bucket-C**: `[/]` items (in-progress — keep state, do nothing)

### Phase 2 — Identify Kanban bindings (per bucket-A)

For each `[x]` Day Planner item, search task surfaces by priority:

1. **Trip-context match**：如 frontmatter `trip-context: [[...]]` 存在 → 读 trip bundle § 1 行程概览 + § 2 议程，匹配 trip-activity items（trip activities 不需进 Kanban）
2. **Explicit wikilink** in Day Planner text → exact wikilink in Kanban
3. **`[chain-anchor]` token** → same anchor in Kanban
4. **Substring containment** (≥6 char, case-insensitive, CN 字符 substring)

Search files (priority):
- `06 Tasks/Today.md` (Today board if exists · renamed from `Tasks.md` 2026-05-21)
- `03 Projects/{{PROJECT_A}}/Tasks.md`
- `03 Projects/{{ORG_B}}/Tasks.md`
- `06 Tasks/Personal.md`

For each `[x]`:
- **Trip-bound** → 不入 Kanban，标 "trip-activity"
- **Kanban-bound** → 准备 propagate `[x]` + update `{{status:: Project.Done}}` + `{{dateCompleted:: ${today}}}` + refresh `{{datetimeModified}}` on matched line
- **Ad-hoc** (no match) → 准备 propose Kanban capture

### Phase 3 — Walk the user (single picker, per ad-hoc 项)

For each ad-hoc `[x]` Day Planner item, show:

```
[x] 11:00 Call with EPIC Tiff (Summit walk-in)
    → not bound to any Kanban / trip activity. Capture?

    (a) Capture to {{PROJECT_A}} Kanban Done (今日已完成的 BD 活动)
    (b) Capture to Personal Kanban Done
    (c) Bind to existing task — search: "Tiff" / "EPIC" 哪个？
    (d) ad-hoc only — 留 Day Planner，不入 Kanban
```

Wait for reply (single letter / 自由)。

Per user CLAUDE.md, **NEVER use AskUserQuestion** — plain text picker。

### Phase 4 — Reverse direction (Kanban → Day Planner)

Scan project surface tasks with:
- `{{dateDue:: ${today}}}` due today
- `{{priority:: 1}}` or `{{priority:: 2}}` (urgent/high)
- `{{status:: Project.Overdue}}` or `{{status:: Project.ThisWeek}}` (lane signals urgency)

Use Grep regex: `\{\{dateDue:: ${today}\}\}` and `\{\{priority:: [012]\}\}` (0=extreme/1=urgent/2=high) and `\{\{status:: Project\.(Overdue|ThisWeek)\}\}`.

For each task NOT already on Day Planner:

```
Project surface has: "{{PERSON}}汇报材料企微发出 + 短会面谈预约" {{dateDue:: 2026-05-14}} {{priority:: 2}}
    → not on today's Day Planner. 加哪？

    (a) 加到 next available time slot (e.g. 13:00)
    (b) 加到 specific time — 几点？
    (c) Push to tomorrow's daily (5/15 carry-forward)
    (d) Skip — handled outside Day Planner
```

### Phase 5 — Apply writes (single batched Edit per file)

Per resolution:

| Resolution | Kanban write | Daily note write |
|---|---|---|
| **Trip-activity** | No project surface change | No daily change (Day Planner 已 `[x]`) |
| **Kanban-bound shipped** | `[ ]` → `[x]` + set `{{status:: Project.Done}}` + `{{dateCompleted:: {today}}}` + refresh `{{datetimeModified}}` on matched line | No daily change |
| **Capture to Kanban** (a/b) | New `[x]` task in surface `## ✅ Done` section with full Operon block (`{{operonId}} {{status:: Project.Done}} {{priority:: 3}} {{dateCompleted:: {today}}} ...`) + 时间 ref `<!-- from Day Planner 11:00 -->` | No daily change |
| **Bind to existing** (c) | `[ ]` → `[x]` + set `{{status:: Project.Done}}` + `{{dateCompleted:: {today}}}` + refresh `{{datetimeModified}}` on bound line | Patch Day Planner line to wikilink the bound task |
| **Ad-hoc only** (d) | No Kanban write | No daily change |
| **Kanban → Day Planner add** (a/b reverse) | No Kanban change | Insert new `- [ ] {time} {task} {{operonId:: <fresh-7char>}} {{datetimeStart:: {today}T{time}:00}}` line in Day Planner, sorted by time (v3 hybrid format — operonId required for Operon indexer to render in Calendar Time Grid) |
| **Kanban → tomorrow carry-forward** (c reverse) | No Kanban change immediate | Append item to today's `**Carry forward to tomorrow:**` section in EOD Review |

### Phase 5b — Reverse direction sweep (Surface-Done → Day Planner-tick)

Added 2026-05-21 PM to close the bidirectional gap: when user marks a surface task Done via Operon UI / Kanban / Calendar during the working day, the matching Day Planner line should also reflect `[x]` so Day Planner OG's "next item" indicator advances correctly.

**Query**: scan Operon indexer for surface tasks with both `{{dateCompleted:: {today}}}` AND a Day Planner counterpart on today's daily note.

```javascript
// Pseudo-CLI via obsidian eval
const op = app.plugins.plugins.operon;
const today = new Date().toISOString().slice(0,10);
const dailyPath = `04 Notes/daily notes/${today}.md`;
const surfaceDoneToday = [];
const dailyOpenItems = [];
for (const [id, t] of op.indexer.tasks) {
  if (t.fieldValues?.dateCompleted === today && t.primary?.filePath !== dailyPath) {
    surfaceDoneToday.push(t);
  }
  if (t.primary?.filePath === dailyPath && t.checkbox === 'todo' /* [ ] */) {
    dailyOpenItems.push(t);
  }
}
// Match each surfaceDoneToday against dailyOpenItems by fuzzy text (≥6 char overlap, CN substring)
// For each match → propose `[ ] → [x]` on Day Planner line + `<!-- bound to {{operonId:: ...}} via 5b sweep -->`
```

**Matching rules** (same priority order as Phase 2 forward direction):
1. **Explicit binding** — Day Planner OR Top 3 line has `<!-- bound: {{operonId:: ...}} -->` comment (written by prior Phase 5 "bind to existing" path OR by `/daily-note` Phase 4 for Top 3 lines) → exact match by operonId. **This is the canonical binding mechanism for Top 3 ↔ surface tasks per `/daily-note` Phase 4 contract.**
2. **Wikilink** — Day Planner text contains `[[...]]` linking to the surface task
3. **Chain-anchor** — both lines share `[chain-anchor]` token
4. **Substring containment** (≥6 char overlap, case-insensitive, CN substring)

**Top 3 line specifics**: when Phase 5b scans daily note, it must scan BOTH `## Day Planner` block AND `## Top 3 Priorities` block. Top 3 lines never have `HH:mm` prefix or `{{datetimeStart}}` field — they're pure narrative bullets with optional `<!-- bound: -->` comment. Skip Top 3 lines with no binding comment (no automated propagation possible — user owns those manually).

**User picker** before any write (same `(a/b/c/d)` shape as forward direction):
```
Surface task ✅ shipped today: "{{PERSON}}汇报材料企微发出 + 短会面谈预约"
  → matches Day Planner 14:00 "{{PERSON}}汇报..."?
  (a) 是 — mark `[x]` on Day Planner 14:00
  (b) 否 — they're different items, skip
  (c) Multiple Day Planner matches: pick one
```

**Write**:
- Day Planner `[ ]` → `[x]` on matched line
- Append `✅ {today}` suffix BEFORE the trailing `{{operonId}} {{datetimeStart}}` block to preserve Day Planner OG's render pattern
- Append `<!-- bound: surface={{operonId:: <id>}} via 5b sweep {{HH:mm}} -->` comment for audit trail

**Conflict avoidance**: Operon's `internalTaskWriteSuppressUntilByPath` machinery already prevents the indexer from re-firing during our writes — leverage it by issuing all 5b writes within a single batched Edit per file. If both the daily note and the surface Kanban are open in Obsidian editors, defer to user-visible warning instead of write conflict.

### Phase 6 — Live frontmatter update

Parse daily note body for new mentions since last sync:

- `cards-spawned` ← scan `[[02 Cards/.../C{date}-...]]` wikilinks created today
- `actions-touched` ← `[[10 Action/.../T{date}-...]]` + chain-anchor tokens  
- `people-touched` ← `[[01 Wiki/.../{Name} ({Org})]]` patterns
- `projects-touched` ← any `[[03 Projects/{name}/...]]` references

De-duplicate + write to frontmatter（preserve order-by-first-appearance）。

### Phase 7 — Summary

```
Synced 2026-05-14 14:32:

Day Planner → Kanban:
  - 2 ad-hoc captures (Tiff conversation, 现场 networking notes)
  - 1 binding ({{PERSON}}汇报 → {{PROJECT_A}} Kanban)
  - 3 trip-activity (no Kanban)

Kanban → Day Planner:
  - 1 added (5/15 退房 prep · 21:00)
  - 0 pushed to tomorrow

Frontmatter:
  - cards-spawned +1
  - actions-touched +1 (沪差0514)
  - people-touched +2 (Tiff, Robin)

Next sync recommended: 18:00 or EOD `--close`
```

## Hard Rules

1. **NEVER auto-toggle `[x]` → `[ ]`** — Day Planner checkbox states are user-authored truth. Sync READS, doesn't reset.
2. **NEVER fabricate `{{dateCompleted:: YYYY-MM-DD}}` stamps** — only on surface task lines that user confirmed shipped (per matched Day Planner `[x]` + user accept binding). Same applies to legacy `✅ YYYY-MM-DD` stamps in pre-Operon files (don't touch).
3. **NEVER batch-capture all ad-hoc items without user picker** — Phase 3 walks each item individually (or surfaces all in one picker block for "reply with letters per item" batch decision).
4. **NEVER use AskUserQuestion tool** — Discord 不渲染 (user global CLAUDE.md)。Plain text picker，等用户回复。
5. **NEVER write to surface `## ✅ Done` section without `{{status:: Project.Done}}` + `{{dateCompleted:: {today}}}` fields** + `<!-- from Day Planner HH:mm -->` provenance comment.
6. **NEVER touch `[/]` in-progress items** — Phase 1 bucket-C does nothing. User changes those manually.
7. **ALWAYS update frontmatter arrays in Phase 6** — without this，mid-day state invisible to other skills（这是 sync 的整个意义）。
8. **ALWAYS respect 09 Rules**: [[09 Rules/tasks.md]] is canonical for Kanban task line format; [[09 Rules/time.md]] for daily note frontmatter contract. Skill defers.
9. **ALWAYS write provenance**: when sync writes to Kanban Done, add `<!-- from Day Planner HH:mm 2026-MM-DD -->` comment line below the task. Future audit / `/vault-evolve` 能追溯。
10. **Day Planner v3 hybrid format (since 2026-05-21)**: 新插 Day Planner 行 → `- [ ] HH:mm task {{operonId:: <7char>}} {{datetimeStart:: YYYY-MM-DDTHH:mm:00}}`。**operonId is REQUIRED** (without it, Operon's indexer ignores the line and Calendar Time Grid won't render it). 读时 strip trailing `{{operonId}} {{datetimeStart}}` block 还原 plain task body。BREAK / END 行不加 Operon block。
11. **Phase 5b reverse sweep is OPT-IN per run**: at Phase 1 user picker, ask once "Run reverse Surface-Done → Day Planner sweep? (y/n)". Default n. The sweep adds latency (full indexer scan + per-match user confirm) and isn't always needed if user only made Day Planner-side changes. When skipped, surface tasks marked Done via Operon UI today are silently OK — Day Planner just won't reflect them until next sync run with sweep enabled OR `/daily-note --close` Phase 7.

## Failure Modes

| Symptom | Fix |
|---|---|
| Today's daily note 不存在 | 停 → 提醒 "今天 daily note 还没建，先 `/daily-note`" |
| Day Planner section 不存在 in daily note | 提醒 "今天 daily 没 Day Planner section — 跳过 Phase 1，只做 Phase 4 Kanban → 加入提议" |
| Multiple Kanban tasks match same Day Planner line (>1 candidate) | Surface all matches，让 user pick (Matt's grill-with-docs 同样 pattern) |
| User wants to undo previous sync's capture | Don't auto-rollback — point to the `<!-- from Day Planner HH:mm -->` provenance line + suggest manual Edit |
| Ad-hoc item user wants to push to tomorrow | Phase 3 选 (d) ad-hoc only + 提醒"明早 Carry forward 自己写一行" OR 在 today's EOD Review § Carry forward 加 |
| Trip-context exists but trip bundle 不存在 | Frontmatter `trip-context:` 指向 dead link — 提醒 user fix link，跳过 trip-context match priority，fallback to wikilink / chain-anchor / substring |
| 出差日 day planner items 大量都是 trip-activity | 正常 — 不 propose 全 capture 到 Kanban（trip activities 本身不是 task），只 surface 极少 ad-hoc 做 capture |
